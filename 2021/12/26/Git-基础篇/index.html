Git-基础篇

git-基础篇

该篇主要分享对 git 初步认识、git 的基础使用以及 git 分支相关操作

对 git 初步认识

  • 我们知道，git 是一个可用来管理代码版本的工具。同类似的工具还有很多，例如 Subversion，相比之下，他俩差异主要是体现在对待数据的方式。svn 是以文件变更列表的方式存储信息，它所存储的信息看作是一组基本文件和每个文件随时间逐步积累的差异，称作是基于差异的版本控制，如图表所示；

  • 而 git 则更像是把数据看做是对小型文件系统的一系列快照，就是说当你提交更新或保存项目状态时，git 基本上会对的全部文件创建一个快照并保存这个快照的索引，对没有修改的文件，git 不会重新存储而只是保留一个链接指向之前的存储文件。简单来说，git 对待数据的方式是直接记录快照，而非差异性比较，git 对待数据更像是一个快照流，如图表所示。

  • git 管理的文件有三种状态：分别是已提交(committed)、已修改(modified) 和 已暂存(staged)。

    • 已修改 表示修改了文件，但还没保存到数据库中。
    • 已暂存 表示对一个已修改文件的当前版本做了标记，使之包含在下次提交的快照中。
    • 已提交 表示数据已安全的保存在本地数据库中

  • git 项目拥有三个阶段：工作区、暂存区以及 git 目录,如图表所示

  • git 的基本工作流程

      1. 在工作区中修改文件。
      1. 将你想要下次提交的更改选择性地暂存，这样只会将更改的部分添加到暂存区。
      1. 提交更新，找到暂存区的文件，将快照永久性存储到 git 目录。

  • 工作目录下文件的状态变化周期：如图表所示

  • .gitignore 忽略文件，可在该文件中指定某种格式的文件，目的是让 git 忽略此文件，不对此文件进行版本控制，不会提交此文件到 git 仓库。忽略文件中的格式举例如图表所示。更加详细的可以查看https://github.com/github/gitignore

  • 分支简介：

    • 上面认识到 git 保存的不是文件的变化或者差异，而是一系列不同时刻的快照

    • 在进行提交操作时，git 会保存一个提交对象，该提交对象会包含一个指向暂存内容快照的指针、作者的姓名和邮箱、提交时输入的信息以及指向它的父对象的指针。首次提交产生的提交对象没有父对象，普通提交操作产生的提交对象有一个父对象， 而由多个分支合并产生的提交对象有多个父对象。对于文件快照，git 使用 blob 对象来存储它们，一个文件快照对应一个 blob 对象。

      举个例子

      • 现在有一个工作目录，里面包含了三个将要被暂存和提交的文件。暂存操作会为每个文件计算校验和（使用一种叫 SHA-1 哈希算法），然后会把当前版本的文件快照保存到 git 仓库，最终将校验和加入到暂存区等待提交。当使用 git commit 进行提交操作时，git 会计算每个子目录的校验和，然后在 git 仓库中这些校验和保存为树对象。随后 git 便会创建一个提交对象，这个提交对象包含指向树对象的指针。现在 git 仓库中有 5 个对象：三个 blob 对象（保存着文件快照）、一个树对象（记录着目录结构和 blob 对象索引）以及一个提交对象（包含者指向前述树对象的指针和所以提交信息） 如图表所示
      • 做些修改后再次提交，那么这次产生的提交对象会包含一个指向上次提交对象（父对象）的指针。如图表所示

git 的基础操作

  • git 的基础操作主要是一些 git 基本命令的使用
    • git init / git clone 这两个命令都可以获取一个 git 项目仓库。git init 是将尚未进行版本控制的本地目录转换为 git 仓库；git clone 从其它服务器 克隆 一个已存在的 Git 仓库。
    • git status / git status -s 这两个命令都可以用来查看当前工作目录下的文件状态,加 -s 参数 可以更简洁的查看文件的状态。如图表所示：
    • git add 用来跟踪新文件以及暂存已修改的文件
    • git diff / git diff -cached/-staged 这两个命令都是用来查看文件快照间的差异。git diff 比较的是工作目录中当前文件和暂存区域快照之间的差异，也就是修改之后还没暂存起来的变化内容。加 -staged 或者-cache 参数就是查看已暂存文件与最后一次提交的文件差异。
    • git commit -m 提交暂存区的文件到 git 仓库 -m 参数后 添加此次提交的操作信息。
    • git rm / git rm –cached git rm 从工作目录中删除指定的文件 git rm –cached 从 git 仓库中删除（亦即从暂存区域移除）指定文件，但仍然希望保留在当前工作目录中。
    • git mv 移动文件 对文件进行重命名
    • git log 查看提交历史
    • git commit –amend 撤销提交操作 有时候我们提交完了才发现漏掉了几个文件没有添加，或者提交信息写错了。 此时，可以运行带有 –amend 选项的提交命令来重新提交。
    • git reset HEAD 取消暂存操作
    • git checkout – 撤销指定文件在本地的任何修改，git 会用最近的版本覆盖掉它。
    • git remote -v 查看远程仓库
    • git remote add 添加远程仓库。该参数表示代替后面整个 URL 作为远程仓库名 相当于给 URL 取个别名 默认是远程仓库名为 origin
    • git fetch 从远程仓库抓取与拉取：这个命令会访问远程仓库，从中拉取所有你还没有的数据。 执行完成后，你将会拥有那个远程仓库中所有分支的引用，可以随时合并或查看。
    • git push 例如 git push origin master 将 master 分支推送到 origin 服务器仓库
    • git push origin 查看某个远程仓库
    • git remote rename 远程仓库的重命名
    • git remote remove/rm 远程仓库的移除
    • git tag -a ‘标签名 例如 v 1.0’ -m “版本一” 为当前版本打一个附注标签
    • git show V1.0 查看标签名为 V1.0 的标签信息和与之对应的提交信息
    • git tag ‘v1.1’ 给提交打轻量标签 轻量标签本质上是将提交校验和存储到一个文件中——没有保存任何其他信息。 创建轻量标签，不需要使用 -a、-s 或 -m 选项，只需要提供标签名字
    • git tag -a v1.2 <某次提交的校验和> 为以前的提交打上标签
    • git push origin 共享标签，将本地的标签 推送到远程服务器上。如果一次性要推送多个标签 可以使用 git push origin –tags 命令。
    • git push origin –delete 删除远程标签
    • git tag -d v.1.1 删除本地轻量标签
    • git config –global alias.co checkout 为 checkout 命令去别名为 co 也就是说在命令行输入 checkout 和 co 效果是一样的

git 分支相关操作

  • git 分支其实就是指向提交对象的可变指针。git 默认分支的名字是 master。 在多次提交操作之
    后，你其实已经有一个指向最后那个提交对象的 master 分支。 master 分支会在每次提交时自动向前移动。

  • git 的 master 分支并不是一个特殊分支。 它就跟其它分支完全没有区别。 之所以几乎每一
    个仓库都有 master 分支，是因为 git init 命令默认创建它，并且大多数人都懒得去改动
    它。

  • 创建一个分支：git branch testing 表示新建一个名为 testing 的分支，这会在当前所在的提交对象上创建一个指针，如图表所示

  • HEAD 分支：它也是一个指针，指向当前所在的本地分支，可以将它想象成当前分支的别名，如图表所示

  • 切换分支：git checkout testing ：这样那个 HEAD 指针就指向 testing 分支了，如图表所示

  • 在当前分支在提交一次 testing 分支向前移动了，但是 master 分支却没有，它仍然指向运行 git checkout 时所指的对象。现在我们切换回 master 分支看看。这条命令做了两件事。 一是使 HEAD 指回 master 分支，二是将工作目录恢复成 master 分支所指向的快照内容。 也就是说，你现在做修改的话，项目将始于一个较旧的版本。 本质上来讲，这就是忽略 testing 分支所做的修改，以便于向另一个方向进行开发。如图表所示

  • 合并分支：git merge testing 在当前分支合并 testing 分支

  • 删除分支：git branch -d testing 删除 testing 分支
